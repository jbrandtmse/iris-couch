## What Didn't Go Well / Challenges

### 1. %request.Content Empty for Multipart/Related Requests
**What happened:** Story 5.2 initially tried to read multipart/related bodies from `%request.Content` and `%Net.MIMEReader`. Both failed — `%request.Content` is empty because %CSP.REST pre-parses multipart bodies into `%request.MimeData`.
**Root cause:** Undocumented %CSP.REST behavior. `MimeData` keys follow an unintuitive naming convention: `CONTENTn` for parts without filename, `FILEn` for parts with Content-Disposition filename.
**Impact:** Extended discovery time for Story 5.2.
**Mitigation:** Saved to project memory for future reference. Document in project rules in Story 6.0.

### 2. Argumented Quit Inside For Loops Causes COMMAND Error
**What happened:** Story 5.1's `Storage.Attachment.Exists()` used `Quit 1` inside a For loop to return early. ObjectScript does not support argumented Quit in For loops — only argumentless `Quit` (which exits the loop).
**Root cause:** ObjectScript's Quit semantics differ between For loops, Try/Catch blocks, and method-level scope. This is the fourth Quit-related gotcha across the project.
**Impact:** Code review caught and fixed it with flag variable pattern.
**Mitigation:** Saved to project memory. Consolidate all Quit restriction patterns in Story 6.0.

### 3. HandleGet Stale Attachment Retrieval After DELETE
**What happened:** Story 5.1's initial `HandleGet` without a `rev` parameter used `FindRevWithAttachment()` which found the attachment on an older revision — even after it had been deleted from the winning revision.
**Root cause:** `FindRevWithAttachment()` searches all revisions, not just the winning one. After DELETE, the winning revision no longer has the attachment, but older revisions still do.
**Impact:** Fixed during development by defaulting to winning revision in HandleGet.
**Mitigation:** Pattern documented in story dev notes.
