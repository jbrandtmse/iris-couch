## What Didn't Go Well / Challenges

### 1. HttpIntegrationTest Behavior Changes Across Stories
**What happened:** TestNotFoundEndpoint and TestParameterizedRoutePattern needed updating in both Story 2.1 (404→405 when PUT/DELETE routes added) and Story 2.2 (405→404 when GET route added). Each new route type changed how unmatched requests were handled.
**Root cause:** Integration tests are coupled to the full route table. Adding routes changes dispatch behavior for existing paths.
**Impact:** Minor — test updates were simple, but it's a pattern that will recur as more routes are added.
**Mitigation:** Consider making integration tests more resilient to route changes by testing specific behavior rather than exact status codes for "any unmatched path" scenarios.

### 2. MakeRequest() Needed Incremental Extension
**What happened:** Story 2.0 created MakeRequest() for GET-only scenarios. Story 2.3 had to extend it with a body parameter for PUT `/_revs_limit`. The response parsing also needed updating to handle plain values (not just JSON objects).
**Root cause:** The helper was designed narrowly for the initial use case.
**Impact:** Minor — extension was clean, but it's a signal that test infrastructure should be designed with future use cases in mind.

### 3. No Automated End-to-End Verification After All Stories
**What happened:** Each story verified its own endpoints via HTTP integration tests, but there was no comprehensive end-to-end test that exercised the full database lifecycle (create → info → list → revs_limit → compact → ensure_full_commit → delete).
**Root cause:** Tests are story-scoped, not epic-scoped.
**Impact:** Low risk — individual endpoints are well-tested, but integration between them isn't verified as a workflow.

## Preparation for Epic 3: Document Storage & Revision Control

### Dependencies on Epic 2

- `Storage.Database.Exists()` — needed before any document operation to verify database exists
- `Storage.Database.Create()/Delete()` — needed for test setup/teardown
- `^IRISCouch.DB(db, "doc_count")` and `^IRISCouch.DB(db, "update_seq")` — need atomic increment on document write
- `^IRISCouch.Seq(db)` — sequence counter for changes feed (initialized in Database.Create)
- Router parameterized route pattern `/:db` — validated and working
- HTTP integration test infrastructure — MakeRequest() with body support ready

### New Capabilities Needed

- `IRISCouch.Storage.Document` — `^IRISCouch.Docs(db, docId, rev)` for document bodies
- `IRISCouch.Storage.RevTree` — `^IRISCouch.Tree(db, docId)` for revision trees
- `IRISCouch.Storage.Changes` — `^IRISCouch.Changes(db, seq)` for changes feed
- `IRISCouch.Core.DocumentEngine` — orchestrator for atomic document writes (body + rev tree + changes + metadata update)
- `IRISCouch.API.DocumentHandler` — CRUD endpoints for `/{db}/{docid}`
- Revision hash generation (deterministic, CouchDB-compatible)
- Optimistic concurrency control (409 Conflict on stale `_rev`)

### Risks

- **Atomic writes (NFR-R2):** Document write must atomically update body, rev tree, changes feed, and database metadata. May need TSTART/TCOMMIT transactions.
- **Revision tree complexity:** CouchDB's revision tree is the most complex data structure in the system. Getting the winning-rev algorithm right is critical (NFR-R1).
- **Global structure design:** The document and rev-tree global structures set here propagate to all subsequent epics. Design carefully.
- **Test complexity:** Document tests require creating databases first, then documents — more setup/teardown complexity.
