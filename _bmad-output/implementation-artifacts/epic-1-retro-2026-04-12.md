## What Didn't Go Well / Challenges

### 1. Catch Block Pattern Confusion (Recurring)
**What happened:** Story 1.2 had `tSC = ex.AsStatus()` after `Error.Render()` already wrote the response. Fixed in review. Same pattern reappeared in Story 1.4's Router.Page(). 
**Root cause:** The interaction between `Error.Render()` and `%CSP.REST`'s status handling wasn't codified clearly after the first discovery.
**Impact:** Two review cycles spent on the same issue.

### 2. %CSP.REST Page() Override Killed Dispatch (Critical — found in manual testing)
**What happened:** Overriding `Page()` in the Router completely replaced `%CSP.REST`'s dispatch mechanism. All HTTP requests hit the 404 fallback instead of the UrlMap routes. Only discovered during live HTTP testing — unit tests calling handlers directly couldn't detect this.
**Root cause:** `%CSP.REST.Page()` IS the dispatch entry point that calls `DispatchRequest()`. Overriding it removes all routing.
**Fix:** Override `ReportHttpStatusCode` for custom error envelopes instead of `Page()`.

### 3. UrlMap Call Attribute Cross-Class Syntax Doesn't Work (Critical — found in manual testing)
**What happened:** `Call="IRISCouch.API.ServerHandler:HandleWelcome"` in the UrlMap silently failed. Routes were never matched.
**Root cause:** Despite documentation suggesting the `ClassName:Method` syntax works, `%CSP.REST` only supports calling methods on the Router class itself.
**Fix:** Create thin wrapper methods on Router that delegate to handler classes.

### 4. Unit Tests Can't Exercise HTTP Layer
**What happened:** All test classes verify `%DynamicObject` structure but none call `Error.Render()` or `Response.JSON()` against a real `%response` object. Deferred in every code review.
**Root cause:** Unit tests run outside the CSP request context — no `%response` object available.
**Impact:** The two critical bugs above (Page override, Call syntax) were invisible to tests.

### 5. ServerHandler Catch Blocks Don't Use RenderInternal
**What happened:** After Story 1.4 added `RenderInternal()` for NFR-S8-compliant 500 error handling, the existing ServerHandler catch blocks still use `Render()`. Exceptions are silently lost instead of being logged to IRIS console.
**Root cause:** ServerHandler was out of scope for Story 1.4's code review.
