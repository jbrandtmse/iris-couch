# Story 3.5: Replication-Format Bulk Writes

Status: done

## Story

As a replication client,
I want to submit writes with `new_edits=false` that preserve source revision IDs,
so that replication can graft external revision histories into the local database.

## Acceptance Criteria

1. **Given** an existing database `{db}`, **When** the client sends `POST /iris-couch/{db}/_bulk_docs` with `{"new_edits":false,"docs":[...]}`, **Then** documents are stored with their provided `_rev` values unchanged, no new revision IDs are generated by the server, and `DocumentEngine.SaveWithHistory()` grafts the external revision history into `^IRISCouch.Tree`.

2. **Given** a document with revision history is submitted via `new_edits=false`, **When** the revision tree is updated, **Then** existing local revisions are preserved, new branches from the source are grafted correctly, and the winning revision is recomputed after grafting.

3. **Given** a `new_edits=false` doc includes `_revisions: {"start":N,"ids":["hash1","hash2",...]}`, **When** the document is stored, **Then** the full ancestry chain is grafted into `^IRISCouch.Tree` with correct R-pointers and leaf/winner nodes.

4. **Given** a `new_edits=false` doc's revision already exists locally, **When** the write is processed, **Then** it is silently accepted (no error, no duplicate data) — this is the idempotency requirement for replication.

5. **Given** all changes are compiled and tested, **Then** all existing ~107 tests pass with zero regressions, and new tests cover all acceptance criteria.

## Dev Notes

### Key Concept: new_edits=false Semantics

In `new_edits=false` mode, the server acts as a passive receiver:
- **No rev minting** — the client provides the exact `_rev` to store
- **No conflict checking** — conflicts are expected and desired (replication creates branches)
- **Ancestry grafting** — the `_revisions` object tells the server where in the tree to place the new revision
- **Idempotency** — if the rev already exists, silently accept it

This is how CouchDB replication works: the source database reads documents with `?revs=true` to get the ancestry chain, then ships them to the target with `new_edits=false`.

### _revisions Object Format

```json
{
  "_id": "doc1",
  "_rev": "3-abc123",
  "_revisions": {
    "start": 3,
    "ids": ["abc123", "def456", "ghi789"]
  },
  "name": "Alice"
}
```

This means:
- Current rev is `3-abc123`
- Parent is `2-def456`
- Grandparent is `1-ghi789`

Reconstruct full rev strings: `start-ids[0]`, `(start-1)-ids[1]`, `(start-2)-ids[2]`, etc.

### GraftChain Algorithm

```
Input: revChain = ["1-ghi789", "2-def456", "3-abc123"]
       leafRev = "3-abc123", deleted = false

For i = 0 to chain.length-1:
  rev = chain[i]
  parentRev = if i > 0 then chain[i-1] else ""
  
  // Skip if R-node already exists (idempotent)
  If $Data(^IRISCouch.Tree(db, docId, "R", rev)) > 0:
    Continue
  
  Set ^IRISCouch.Tree(db, docId, "R", rev) = parentRev

// Handle leaf: if parent was a leaf before, it's no longer one  
parentOfLeaf = chain[chain.length-2] if length > 1
If parentOfLeaf has "L" entry:
  Kill ^IRISCouch.Tree(db, docId, "L", parentOfLeaf)
  Kill ^IRISCouch.Tree(db, docId, "D", parentOfLeaf)

// Set new leaf
Set ^IRISCouch.Tree(db, docId, "L", leafRev) = chain.length
If deleted:
  Set ^IRISCouch.Tree(db, docId, "D", leafRev) = 1

RecomputeWinner()
```

### new_edits=false Response Format

Per CouchDB spec, `new_edits=false` returns `201` with an empty JSON array `[]` on success. Individual document errors are not reported — the replicator handles retries.

### doc_count Tracking for new_edits=false

Only increment `doc_count` if this is a completely new document (no prior R-nodes existed for this docId). Check `$Data(^IRISCouch.Tree(pDB, pDocId))` before grafting — if 0, it's a new doc.

### No _revisions Fallback

If a `new_edits=false` doc has `_rev` but no `_revisions`, treat it as a single-node tree: the document is the root and only revision. This is less common but valid.
