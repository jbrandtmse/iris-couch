## What Went Well

1. **The Task 0 rule held across twelve consecutive stories.** Codified in Story 12.0, applied in every subsequent Epic 12 story. It stopped being "the new rule" and became how the team works. The probe-first discipline caught the `$ZF(-100)` file-redirection decision in 12.2 before a single line of Pipe code was written, and caught the Python unavailability in 12.4 before any Python class was authored.

2. **~8000× view-query speedup is real and measured, not marketing.** 1288 ms per-query cold-spawn (Story 12.2 baseline) → 0.16 ms `$Order` index walk (Story 12.5 benchmark). Byte-identical JSON output across pre/post harness. The architecture choice to index at write time rather than query time turned a non-shippable latency profile into a production-ready one in a single story.

3. **Story 12.5 closed four MEDs while adding new capability.** NFR-S9 sandbox hardening (Story 12.2 deferral), `JSRUNTIMETIMEOUT` enforcement (Story 12.2 deferral), `ListValidateFunctions` O(N) iteration (Story 12.3 deferral), per-change filter subprocess spawn cost (Story 12.3 deferral) — all resolved by the incremental-indexing + pool-API architecture. This is the first epic in recent memory where a single story *shrank* the deferred-work file.

4. **Adversarial code review caught the Pool wiring gap in 12.5.** Dev's first pass shipped `Subprocess.StartPipe` calling `Pipe.%New()` directly, bypassing the Pool API. AC #7 had promised Pool usage. The reviewer auto-resolved by routing `StartPipe` through `Pool.Acquire()` / `Release()`. Observable behaviour today is unchanged (Pool is a shim), but when Story 12.5b lands, the true persistent pool is a drop-in replacement rather than a refactor. That is the adversarial review layer earning its keep on *architectural invariants*, not just code style.

5. **Explicit scope discipline in Story 12.2.** Rather than ship partial view-query parameter support, dev-12-2 cut scope explicitly to Story 12.2a (`group`, `group_level`, `startkey`, `endkey`, `limit`, `skip`) and logged it cleanly. The subsequent 12.5 work built on a crisp MVP rather than a half-done feature.

6. **Story 12.4 deferral was the *right* call, made at the right time.** Dev hit the IRIS CPF Python-runtime blocker at Task 0, reverted every change, escalated cleanly. Lead + user decided to defer explicitly rather than silently skip tests or ship a stubbed Python.cls into production. README updated in the same commit, Subprocess/Node named as the sole α/β JS path, and the gap honestly documented.

## What Didn't Go Well / Challenges

### Story 12.4 — Python blocker exposed a pre-flight gap

The story spec assumed IRIS embedded Python was available; Task 0 discovered it wasn't. The dev environment's `PythonRuntimeLibrary` and `PythonRuntimeLibraryVersion` CPF fields were blank, the only installed Python (3.13) was rejected by IRIS for missing PE VERSIONINFO, and `%SYS.Python.Import("sys")` threw. This forced a full revert and epic-level deferral decision.

**What this teaches us:** the `/bmad-create-story` workflow produces excellent spec-level Task 0 probes, but it writes 300+ lines of acceptance criteria and tasks *before* the first environment check runs. When a story depends on a runtime capability the dev host may lack, the environment check should be the *first* output of story creation — before the spec is fleshed out.

**Preventive measure (Action Item #1):** pre-flight dev-environment capability check at story-creation time for any backend-integration story. Confirms the dev host can exercise the core capability before the spec gets authored.

### Distribution-level compile risk not flagged in 12.4 story spec

The retrospective surfaced (via Josh) that if Story 12.4 had *succeeded* on a Python-enabled dev host, it would likely have shipped `[Language = python]` methods in IRISCouch classes. Those methods would fail to compile at install time on any Python-less IRIS instance — the entire `zpm "install iris-couch"` command would abort.

**This is the single highest-value finding of this retrospective.** The story spec did not flag compile-time Python dependency as a risk. If dev-12-4 had completed on a Python-enabled host, we would have shipped an install-broken package to every customer whose IRIS build doesn't include Python.

**Preventive measures (Action Items #6–#9):**
- Zero `[Language = python]` methods in any shipped `.cls` file (release-gate rule when 12.4 resumes)
- Python bridge (`jsruntime.py`) ships as ZPM `<FileCopy>` resource, never embedded in a class
- `irispip install quickjs` is documented as an operator prerequisite, never invoked from a ZPM install hook
- A Python-less IRIS CI image runs `zpm install iris-couch` as a release gate

### Story 12.2 scope cut was silently absent from README until this retro

Dev-12-2 explicitly cut view-query parameters (`group`, `group_level`, `startkey`, `endkey`, `limit`, `skip`) to Story 12.2a. The cut was documented in the story file, in `deferred-work.md`, and in the commit message — but *not* in the README. Operator-facing documentation claimed `Subprocess` supports view queries without qualifying which parameters. Only caught because Josh surfaced it in the retro discussion.

**Preventive measure (Action Item #2):** when a story is deferred due to environment or scope, update `README.md` in the same commit as the sprint-status flip. The pattern was applied correctly for 12.4's deferral but missed for 12.2's scope cut.

### Story 12.5 `Pool.cls` class docstring over-describes the implementation

The class docstring describes a LIFO/MRU stack layout backed by a PPG with per-entry timestamps. The implementation is a shim that always spawns fresh — real pool behaviour lives in Story 12.5b. Code review flagged this as LOW and the docstring should be tightened to "Planned for 12.5b — shim today." Minor cosmetic issue, but it's the kind of thing that lies to the next reader.

### MCP `iris_execute_tests` class-level discovery inconsistency (carried from 12.1)

Running `iris_execute_tests` at `level=class` against the Epic 12 test classes (JSRuntimeHttpTest, JSRuntimeSubprocessHttpTest, JSRuntimeValidateHttpTest, JSRuntimeFilterHttpTest, ViewIndexHttpTest) reports fewer test methods than are defined. Running at `level=method` discovers them all and produces correct counts. Workaround: bespoke test runners (`SubprocessTestRunner`, `SubprocessValidateProbe`) that invoke methods directly via `$METHOD` and walk `^UnitTest.Result`. Reproducible across multiple stories; should be filed against `iris-dev-mcp` when time permits. No product impact.

## Key Insights

1. **The Task 0 rule compounds.** Twelve consecutive stories have applied it; the cost per story is low because probe commands become muscle memory, and the catch rate is real — 12.4's blocker was caught before any code, 12.2's `$ZF(-100)` decision was made before any Pipe code. The rule's value isn't what it catches in any single story; it's that the architectural decisions get grounded in verified evidence rather than optimistic assumption.

2. **Adversarial code review is the right layer to catch architectural invariants.** Three MEDs auto-resolved in 12.5 review (Storage encapsulation, body-rewrite Pattern Replication gap, Pool wiring) all required understanding what the *code promised* vs what it actually did. A syntactic linter or test suite can't catch "this class's docstring claims a LIFO stack but the implementation spawns fresh." A human (or adversarial-review agent) reading the code and cross-checking against the ACs can.

3. **Shim-with-real-API-shape is a better architecture than stubbed-signature.** Story 12.5's `Pool` class exposes the full `Acquire`/`Release`/`Reap`/`ShutdownAll` surface and every caller goes through it — even though today `Acquire` just spawns fresh. When the true persistent implementation lands in 12.5b, no caller changes. Contrast with Story 12.1's None backend, which threw `not_yet_implemented` at call sites and required those sites to be rewritten in 12.2 and 12.3. Shim-with-API-shape wins on change velocity.

4. **README is documentation for operators, not for the dev team.** The 12.2 scope-cut invisibility to Josh revealed that the dev team tracks internal state via story files and `deferred-work.md`, but operators only see the README. Any story deferral, scope cut, or known limitation that an operator could observe *at runtime* must land in the README in the same commit. This is now an explicit rule.

5. **"The epic we defer is as important as the epics we ship."** Story 12.4's deferral, combined with Josh's distribution-risk observation, produced three architectural rules for when 12.4 resumes (zero `[Language = python]`, file-copy Python bridge, operator-prerequisite `irispip`) plus a new NFR candidate for the PRD (compile-on-any-IRIS). These rules are Epic 12 outputs; they didn't require Epic 12 to succeed at 12.4 to be valuable.
