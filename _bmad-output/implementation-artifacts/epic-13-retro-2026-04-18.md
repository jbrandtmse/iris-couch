## What Didn't Go Well / Challenges

### Documentation drift surfaced as "invented endpoints" — the hardest failure mode to catch

Epic 13 is the project's **first pure-documentation epic**. Every prior epic had an ObjectScript or Angular "is the code correct?" question at the center, answered by test suites. Epic 13's core question is "is the documentation complete and accurate?" — and that question has no automated answer. Three of the four stories had reviewer-caught documentation bugs where the author cited endpoints/tests/globals that *did not exist*:

- **13.1** cited ~45 test-method names that had been renamed or never existed (`DocumentApiTest.TestPostCreate` vs the actual test method name, etc.)
- **13.1** missed 7 CouchDB 3.x endpoint families entirely from the matrix, and miscategorized `/{db}/{docid}` COPY as `supported` when no handler or test exists
- **13.2** cited `_metrics` (real route is `_prometheus`), `_node/_local/_config/jsruntime` (OOS — doesn't exist), `^IRISCouch.Audit(<seq>)` (global doesn't exist; audit goes to `%SYS.Audit`)
- **13.2** invented `_local/_replicator_checkpoint_<session_id>` as the checkpoint-doc pattern when the actual CouchDB spec is `_local/{replication-id}` (a hash)

**What this teaches us:** LLM dev agents, when writing documentation, fill in gaps with plausible-sounding content. A stale test-method name "sounds like" a real test-method name. An endpoint that "feels" like it would exist because it fits the URL pattern gets cited as if it does. These are not hallucinations in the strict sense — they're *plausibility extrapolation*, and they are invisible to the author because the author isn't re-probing every claim.

**Preventive measure (Action Item #1):** every documentation story must include a **documentation-verification Task** analogous to "run the test suite" for code stories. The verification walks every endpoint/test-method/global/config-key claim in the new document and greps for it in the source tree. Any claim that doesn't resolve is either (a) struck from the doc, (b) accompanied by a "not yet shipped, planned for Story X" pointer, or (c) escalated as a matrix correction. This is what the Story 13.2 reviewer did after-the-fact — it needs to run *before* review.

### The lead's pre-authored probe blocks were speculative, not verified

For Story 13.1 and Story 13.3, the lead orchestrator pre-authored Task 0 probe blocks ("curl /iris-couch/ returns version: 3.3.3") into the story file so the dev agent had a capture template to match. Both times the dev agent discovered the actual output differed (server returns `0.1.0`, no `features` array) and correctly substituted per the story's own instructions — but the lead's speculation *became* misinformation that any less-disciplined dev agent could have copy-pasted straight into the shipped doc.

**Preventive measure (Action Item #2):** the lead orchestrator should not pre-author probe captures in story files. Instead, the lead runs the probes during story-creation (as part of the research-first.md Task 0 rule applied *to story creation itself*) and pastes the verbatim output. If the lead is working at a distance from the dev host (e.g., reviewing on a different machine), mark the Task 0 block as "dev agent to capture" and leave it empty — better to have a hole than a plausible fiction.

### Story 13.2's invented-endpoint density was higher than Story 13.1's

Story 13.1 had 2 MED findings (stale test-method citations + missed endpoint families); Story 13.2 had 3 HIGH + 1 MED, all invented endpoints. The contributing factor: Story 13.2's three docs are *instructional* (diagnostic commands, migration steps, troubleshooting recipes), which require the dev agent to *imagine* the operator running the commands. Imagination without verification produces plausible fiction. Story 13.1 was more *referential* (here's the list of endpoints, here's their status) — and referential docs have fewer opportunities to confabulate executable content.

**Preventive measure (Action Item #3):** for instructional docs specifically, every code-block in the doc (every `curl`, every `iris session` command, every `zwrite`) must be executable on the dev host and its output captured. The reviewer should run a random sample during review. Story 13.3 did this naturally — because examples *are* runnable code — which is why 13.3 shipped clean. Instructional markdown should be held to the same standard.

### The trailing-slash routing bug is a real, user-visible PouchDB integration failure

Discovered during Story 13.3 when PouchDB's default constructor hit `PUT /{db}/` (trailing slash) and IRISCouch returned 404 because `src/IRISCouch/API/Router.cls` declares `<Route Url="/:db" ...>` but no `/:db/` companion routes. CouchDB 3.x accepts both. This is an **α-blocking integration bug** — every adopter who does `new PouchDB('http://host/iris-couch/dbname')` without the `skip_setup: true` option will see the replication fail silently with a 404.

**Preventive measure (Action Item #4):** fix in the next backend cleanup story (`14-0-epic-13-deferred-cleanup.md`). Four-line Router.cls change; same handler delegation. Escalated as HIGH in `deferred-work.md`. Also: any backend story touching Router.cls must now audit CouchDB client tolerance patterns (trailing-slash, case-sensitive paths, etc.) — the bug was invisible to every prior epic because none of them integrated with a real production PouchDB client.

### Examples CI is unwired — manual-release-gate is single-point-of-failure

Story 13.3 shipped `examples/run-all.sh` + `examples/run-all.ps1` but chose Option B (defer CI wiring). The current release-gate enforcement is "the author runs run-all.sh on the dev host before tagging" — which works until it doesn't. The Story 11.0 UI smoke workflow hit the same wall (self-hosted-runner-required, queued indefinitely).

**Preventive measure (Action Item #5):** before α/β tagging gate, ship a dockerized IRIS CI image + GitHub Actions workflow(s) that run both `ui-smoke.yml` and a new `examples-smoke.yml` against a provisioned IRIS. Same infra story resolves both deferrals. Scoped as a standalone infra story, not bundled with Epic 14 feature work.

### The `/_node/_local/_config/jsruntime` non-endpoint nearly shipped three times

The Story 13.0 rules explicitly cited `/_node/_local/_config/jsruntime` as the diagnostic surface for verifying `JSRUNTIME` config. Story 13.1 matrix listed it OOS (correctly). Story 13.2 troubleshooting runbook cited it as a diagnostic command (incorrectly — OOS means it doesn't work). Story 13.3 examples setup.js used an observable probe instead (correctly — dev agent read the matrix). The endpoint pattern "I thought there was a config endpoint" appeared in 13.0, 13.2, and was almost reintroduced in 13.3. **There is no HTTP surface for reading IRISCouch config today.** Operators must read the `^IRISCouch.Config` global via `iris session` or the Management Portal.

**Preventive measure (Action Item #6):** consider whether shipping a read-only `GET /_node/_local/_config/<key>` endpoint is worth the scope for α/β — the repeat-offender pattern suggests operators will expect it. Either ship it, or document its absence more aggressively (a deviation entry, not just an OOS matrix row).

## Key Insights

1. **Pure-documentation epics have a different failure mode than code epics.** Code epics fail "feature doesn't work" and tests catch it. Documentation epics fail "doc describes a feature that doesn't exist in the shape described" and nothing catches it except adversarial review. The Epic 13 pattern — 8 MEDs+higher auto-resolved across four stories, all documentation-drift — is not a sign of weak dev work; it is a sign that documentation verification needs its own Task-level discipline. The "run the test suite" equivalent for docs is "grep every claim."

2. **Shim-with-real-API-shape applies to documentation too.** Story 13.3's `examples/run-all.sh` is a shim for the eventual CI workflow (Option B deferral). The harness has the real output contract — exit codes, summary format, per-example pass/skip — so when CI wiring lands, the workflow calls the existing script unchanged. Contrast with "we'll write the runner when we wire CI" which would have left the wiring as greenfield. Same lesson as Story 12.5's Pool API shim: ship the contract today, real implementation tomorrow, no caller changes.

3. **The dev-agent-reads-the-source-tree discipline outpaces the dev-agent-reads-the-spec discipline.** Story 13.3's dev agent caught the view-param matrix lie *by probing the live instance*, not by reading the story spec. Story 13.2's dev agent invented endpoints *by writing from the spec's suggested commands*. The lesson: specs can contain errors (lead is fallible); source code and live instance output cannot lie. For every documentation story, the dev agent should prefer a probe over a spec citation.

4. **"Self-application" is a legitimate validation technique for new rules.** Story 13.0 codified two new process rules (pre-flight capability check + README-same-commit). Every subsequent Epic 13 story applied them. By end of epic, the rules have four application data points, and both are demonstrably helpful. If either rule had caused friction or produced false positives, the same epic that introduced them would have exposed the problem. Codify-new-rule then apply-immediately-in-same-cycle is a faster feedback loop than codify-now, observe-next-epic.

5. **The release-gate commitment "broken example blocks release" is only as strong as the enforcement mechanism.** Shipping `run-all.sh` without CI wiring technically satisfies AC #4's Option B, but it leaves enforcement dependent on author vigilance. Every release-gate commitment should be accompanied by a question: "What stops this from rotting if the committer forgets?" For Epic 13: the answer today is "nothing." For Epic 14: the answer must be CI.

6. **Epic 13 is where the project decided to tell the truth.** Every deviation the `deviations.md` audit surfaced — mixed-type collation, `_approx_count_distinct` exact-count, view-param 12.2a scope cut, Python deferred, server version `0.1.0`, `_security` shape, CSP Gateway buffering, Windows memory cap, render-endpoint 404-not-501, `/_utils/` Angular vs Fauxton — is a thing that *could have been minimized or spun* in a marketing-tilted README. None of them were. The product's truthfulness posture is now *codified in artifacts that ship with every release*. That is a higher bar to maintain than any single release, and it is the project's most durable competitive advantage against the "CouchDB-compatible" products that overstate compatibility.
