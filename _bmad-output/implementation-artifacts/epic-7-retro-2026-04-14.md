## What Went Well

1. **Comprehensive auth subsystem delivered in one epic** — 4 auth mechanisms (cookie, JWT, proxy, basic), _users database with PBKDF2 sync, per-database _security enforcement. The thin-handler architecture from earlier epics scaled cleanly to the auth domain.

2. **CouchDB source reading proved critical** — Reading `jwtf.erl`, `couch_httpd_auth.erl`, `couch_users_db.erl`, and `security.rst` directly gave exact algorithm specs (HMAC-SHA1 for proxy, HS256 JWT format, password field stripping, _security semantics). More reliable than external docs alone.

3. **Code review auto-resolve caught real security issues** — Timing-safe HMAC comparison, AUTHSECRET exposure in GetAll(), future-dated cookie rejection, namespace restore in catch blocks, MangoIndex re-index after body modification. 7 HIGH/MEDIUM issues auto-resolved that would have been production vulnerabilities.

4. **OnPreDispatch architecture clean** — Auth context establishment (cookie > bearer > proxy > basic > anonymous) and _security enforcement both run before any handler code, exactly as NFR-S6 requires. Denied requests never reach storage.

5. **Manual curl testing caught the critical Login() bug** — The 37-test manual verification suite found that `$System.Security.Login()` switches process context, breaking all non-admin users. Automated tests missed this because they all run as _SYSTEM. This validates the "always verify via curl" pattern for the 7th consecutive epic.

## What Didn't Go Well / Challenges

### 1. $System.Security.Login() Switches Process Context (CRITICAL BUG)
**What happened:** All non-admin users got 500 errors on every endpoint. `$System.Security.Login()` doesn't just validate credentials — it switches the entire process to run as that user, who lacks `%DB_IRISCOUCH` namespace access.
**Root cause:** The IRIS documentation for `Login()` says "Log in a user" but doesn't emphasize that it's a destructive context switch. We didn't read the actual source at `irislib/%SYSTEM/Security.cls` before using it.
**Fix:** Replaced with `Security.Users.CheckPassword()` which validates without context switch. Also added infrastructure roles (`%DB_IRISCOUCH`, `%DB_IRISLIB`, `%DB_IRISTEMP`) when creating users.
**Impact:** Required Story 7.5 hotfix via correct-course process. All non-_SYSTEM auth was broken until fixed.
**Lesson:** **Always read IRIS library source (`irislib/`) before using unfamiliar IRIS APIs.**

### 2. Native PBKDF2 Existed But Was Manually Reimplemented
**What happened:** The dev agent for Story 7.3 reported `$System.Encryption.PBKDF2()` doesn't exist and built a 30-line manual implementation via iterated HMACSHA.
**Root cause:** The agent likely got an error when testing (possibly wrong parameter order) and concluded the method doesn't exist, rather than reading the actual source at `irislib/%System/Encryption.cls:1111`.
**Fix:** Replaced with single-line `$System.Encryption.PBKDF2(password, iterations, salt, keyLength, 256)`.
**Lesson:** Same as #1 — read `irislib/` source. The method exists and works.

### 3. Subagent Not Told About CouchDB Source
**What happened:** The Story 7.2 dev agent didn't know about `sources/couchdb/` until the user reminded us. The initial research task used only Perplexity, missing exact algorithm details.
**Fix:** Added memory `feedback_couchdb_source_first.md` and `reference_couchdb_source.md`. All subsequent agents were briefed.
**Lesson:** **Always include CouchDB source and IRIS library source references in subagent prompts.**

### 4. IRIS HMACSHA Uses Bit Sizes, Not Version Numbers
**What happened:** `$System.Encryption.HMACSHA(1, ...)` threw `<ILLEGAL VALUE>`. The correct call is `HMACSHA(160, ...)` for SHA-1.
**Root cause:** IRIS uses bit lengths (160, 256, 384, 512) not algorithm version numbers.
**Fix:** Changed to `HMACSHA(160, ...)` for proxy auth SHA-1 and `HMACSHA(256, ...)` for JWT SHA-256.
**Lesson:** Reading `irislib/%System/Encryption.cls` would have shown the parameter docs immediately.
